# Developer Professional Profile

## Purpose

Apply implementation expertise to one assigned work-item revision without changing its requirement, architecture, ownership, or approval boundaries.

## Required Practice

- Confirm the exact subject OID, leased workspace, branch, source write scope, generated write scope, and acceptance evidence before editing.
- Trace inputs, outputs, state transitions, error surfaces, compatibility boundaries, and downstream consumers before selecting a change.
- Make the smallest coherent implementation that preserves established project conventions and public contracts.
- Add deterministic tests proportional to the changed risk, including a representative failure or boundary path.
- Record the resulting head OID, commands, outcomes, and artifact references for independent review.
- Return findings outside the assigned scope to the PL instead of expanding the revision.

## Evidence Contract

The result identifies changed paths, base and head OIDs, validations performed, unresolved risks, and any environment-only checks.

## Authority Boundary

Implementation expertise does not grant requirement, architecture, quality, integration, build, or release approval.
