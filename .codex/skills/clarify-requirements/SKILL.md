# Clarify Requirements

Turn incomplete requests into evidence-labeled, implementation-ready requirements without inventing product intent.

## Procedure

1. State the user or business outcome in observable terms.
2. Separate confirmed requirements from repository evidence, assumptions, and open decisions.
3. Define in-scope, out-of-scope, and deferred behavior.
4. Record technical, operational, compatibility, and delivery constraints.
5. Write testable acceptance criteria, including failure and boundary behavior.
6. Assign every unresolved decision to the organizational role that owns it.

## Quality Rules

- Map every requirement to a concrete behavior or outcome.
- Preserve explicit user language when it defines a hard constraint.
- Surface contradictions instead of silently selecting one interpretation.
- Do not convert an assumption into an approval-ready requirement.
- Prefer the smallest scope that achieves the stated outcome.

## Return Contract

Return:

- problem statement and desired outcome;
- confirmed scope, non-goals, and deferred work;
- acceptance criteria;
- constraints and dependencies;
- assumptions with evidence labels;
- open decisions, decision owner, and blocking impact.

## Authority Boundary

This skill supplies analysis expertise only. It does not grant product authority, approval rights, task ownership, model selection, tool permissions, or permission to edit files.
